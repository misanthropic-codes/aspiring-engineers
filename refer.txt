SYSTEM OVERVIEW
================================================================================

The Refer & Earn system allows users to:
1. Share unique referral codes with friends
2. Earn "points" when friends signup using their code
3. Earn bonus points when referred friends make their first purchase
4. View referral history and points transactions
5. Redeem points for discounts (future integration with checkout)

BUSINESS RULES:
- Referred user gets 10 pts immediately on signup
- Referrer gets 50 pts when referred user makes first successful payment
- Self-referral is not allowed
- One referral code per user (auto-generated on first login)
- Referrals become "rewarded" only after first successful payment
- Admins can disqualify referrals for fraud with point recovery

================================================================================
AUTHENTICATION & REGISTRATION FLOW  
================================================================================

1. USER SIGNUP WITH REFERRAL CODE
   =================================
   POST /auth/register
   
   Request:
   {
     "email": "newuser@example.com",
     "password": "SecurePass123!",
     "name": "John Doe",
     "phone": "+919876543210",
     "dateOfBirth": "1995-05-15",
     "examTargets": ["NEET", "JEE"],
     "targetYear": 2025,
     "referralCode": "AEAB12CD"  ← OPTIONAL - referrer's code
   }
   
   Response (201 Created):
   {
     "success": true,
     "message": "Registration successful! Please check your email for the 6-digit verification code.",
     "data": {
       "email": "newuser@example.com",
       "name": "John Doe"
     }
   }
   
   FLOW:
   1. New user signs up with email, password, and optional referralCode
   2. System validates the referralCode if provided
   3. If valid, creates a PENDING referral record linking:
      - Referred user ID
      - Referrer user ID
      - Referral code used
   4. New user is awarded 10 pts immediately (referred_bonus)
   5. Referred user must verify email (6-digit OTP) to complete registration
   
   ERRORS:
   - 400: referralCode not found → "Invalid referral code"
   - 400: Self-referral → "Self referral is not allowed"
   - 400: Already has referrer → "Referral already linked for this user"

TROUBLESHOOTING: WHY IS REFERRAL CODE NULL?
==============================================

Symptom: GET /referrals/me/summary returns referralCode: null

Root Causes:
  1. NEW USER BEFORE REFERRAL SYSTEM
     - User registered before referral module was deployed
     - Database records don't have referralCode field set
     
  2. REGISTRATION FLOW INTERRUPTED
     - Email verification failed during signup
     - User created but ensureUserReferralCode() didn't complete
     - Network error after user creation but before code generation
     
  3. OLD DATABASE FROM PRODUCTION
     - Migrated database has users without referral codes
     - Not initialized during original signup flow

Solution:
  - Call POST /referrals/me/generate-code (as the user)
  - System will auto-generate unique 8-char code
  - Code stored and available in next GET /referrals/me/summary

Example:
  POST /referrals/me/generate-code (with valid JWT)
  →
  {
    "success": true,
    "message": "Referral code generated successfully",
    "data": {
      "referralCode": "AEAB12CD"
    }
  }

================================================================================
POINTS ALLOCATION RULES
================================================================================

REFERRAL INITIATION:
  - Trigger: User signs up with valid referralCode
  - Referred user: +10 pts (type: referred_bonus)
  - Referrer: Stays pending until first purchase
  - Status: PENDING

FIRST SUCCESSFUL PAYMENT:
  - Trigger: Payment webhook from Cashfree with PaymentStatus.SUCCESS
  - Referred user: Already has +10 pts (no additional)
  - Referrer: +50 pts (type: referrer_bonus)
  - ReferralTracking.status: Changed to REWARDED
  - Idempotency: Prevents double-crediting via unique idempotencyKey
  
  Idempotency format:
    - Referrer bonus: "referrer_bonus:{trackingId}:{referrerUserId}"
    - Referred bonus: "referred_bonus:{trackingId}:{referredUserId}"

POINTS DEDUCTION:
  - Type: redemption_debit (usually -10, -50, etc.)
  - Trigger: User redeem points for discount/product
  - Impact: pointsBalance decreased, totalPointsRedeemed increased

ADMIN ADJUSTMENTS:
  - Type: admin_adjustment
  - Referrer can get +50 pts recovered if referral disqualified
  - Idempotency: "admin_disqualify_recovery:{referralId}:{referrerUserId}"


================================================================================
IMPLEMENTATION DETAILS
================================================================================

1. SIGNUP FLOW WITH REFERRAL CODE
   ================================
   a) User calls POST /auth/register with optional referralCode
   b) Auth service validates email/phone not already registered
   c) If referralCode provided:
      - ReferralsService.validateReferralCode() checks code exists
      - Throws error if code not found or is user's own code
   d) User created in database with email verification code
   e) Verification email sent to user
   f) Referral code auto-generated for new user
   g) If valid referralCode provided:
      - ReferralsService.createPendingReferral() called
      - ReferralTracking created with status=PENDING
      - Referred user gets +10 pts immediately
   h) Return success message (tokens not given until email verified)

   KEY: Referral happens BEFORE email verification completes


2. PAYMENT SUCCESS HOOK
   ====================
   a) User makes payment via Cashfree payment gateway
   b) Payment status = SUCCESS
   c) Webhook/Payment processing calls paymentService.handlePaymentSuccess()
   d) paymentService calls referralsService.handleSuccessfulPurchase(userId, paymentId)
   e) ReferralsService checks:
      - Is there a PENDING referral for this userId?
      - Has user made their first payment? (check payment count)
   f) If yes:
      - ReferralTracking.status = REWARDED
      - Referrer gets +50 pts
      - Referred user gets +10 pts (if not already awarded)
      - Both transactions created with idempotency keys
   g) If referral already processed: Skip (idempotency check)


3. REFERRAL CODE GENERATION
   =========================
   - Format: 8 characters (2 prefix: "AE" + 6 random)
   - Alphabet: ABCDEFGHJKLMNPQRSTUVWXYZ23456789 (no I, O, L, 0, 1 to avoid confusion)
   - Example: AEAB12CD
   - Uniqueness: Ensured by MongoDB unique index
   - Auto-generated: On first login (ensureUserReferralCode)
   - Non-transferable: One code per user lifetime


================================================================================
IDEMPOTENCY & DOUBLE-SPEND PREVENTION
================================================================================

Problem: Payment webhook can be retried multiple times
        User might accidentally make duplicate requests
        
Solution: idempotencyKey field in PointsTransaction schema

Flow:
  1. System generates unique idempotencyKey for each transaction
  2. Before crediting points, check if idempotencyKey already exists
  3. If EXISTS: Skip (already processed)
  4. If NOT EXISTS: Create transaction and credit points

Idempotency Key Formats:
  - referrer_bonus: "referrer_bonus:{trackingId}:{referrerUserId}"
  - referred_bonus: "referred_bonus:{trackingId}:{referredUserId}"
  - admin_adjust: "admin_adjust:{userId}:{timestamp}"
  - disqualify_recovery: "admin_disqualify_recovery:{referralId}:{referrerUserId}"


================================================================================
SECURITY CONSIDERATIONS
================================================================================

1. SELF-REFERRAL PREVENTION
   - System checks: referrerUserId !== referredUserId
   - Error: "Self referral is not allowed"

2. DUPLICATE REFERRAL PREVENTION
   - Each user can only be referred once
   - DB unique index on referredUserId in ReferralTracking
   - Error: "Referral already linked for this user"

3. POINTS CANNOT BE NEGATIVE
   - Update operations use idempotency to prevent overwrites
   - Redemption validates sufficient balance before debit
   - Admin adjustments explicitly validate amounts

4. JWT PROTECTION
   - All user endpoints require valid JWT token
   - Token contains userId for authorization
   - Endpoints only return data for authenticated user

5. FRAUD DETECTION (ADMIN ONLY)
   - Suspicious pattern detection
   - Email/phone matching between referrer and referred
   - Fast conversion detection (< 6 hours)
   - Risk scoring and recommendations
   - Manual disqualification with point recovery
